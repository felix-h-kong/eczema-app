# Eczema Trigger Tracker — Design Spec

## Overview

A personal PWA for logging every meal, flare-up, and medication, then statistically correlating ingredients with eczema flares over time. Self-hosted on a home laptop, accessed via Tailscale VPN from an Android phone.

## User Context

- Location: Sydney, Australia (AEST/AEDT)
- Device: Android phone
- Hosting: Dedicated always-on home laptop + Tailscale
- Privacy: Health data never leaves own infrastructure unencrypted
- Goal: Log every meal daily, correlate ingredients with flares over weeks/months

## Data Model

All timestamps stored in UTC (ISO 8601, e.g. `2026-03-23T08:30:00Z`). Frontend converts to local time for display using the device's timezone.

```sql
CREATE TABLE log_entries (
    id INTEGER PRIMARY KEY,
    timestamp TEXT NOT NULL,
    type TEXT NOT NULL,                -- 'meal' | 'flare' | 'medication' | 'note'
    raw_input TEXT,
    parsed_ingredients TEXT,           -- JSON: {"confirmed":[], "likely":[], "source":"text"|"image"}
    parse_status TEXT DEFAULT 'pending', -- 'pending' | 'parsed' | 'failed'
    severity INTEGER,                  -- 1-10, flare only
    medication_name TEXT,
    medication_dose TEXT,
    notes TEXT,
    synced INTEGER DEFAULT 0
);

CREATE TABLE ingredient_aliases (
    id INTEGER PRIMARY KEY,
    variant TEXT UNIQUE,
    canonical TEXT
);

CREATE TABLE entry_images (
    id INTEGER PRIMARY KEY,
    log_entry_id INTEGER REFERENCES log_entries(id),
    image_path TEXT,                   -- relative path within data/images/
    timestamp TEXT
);

CREATE TABLE push_subscriptions (
    id INTEGER PRIMARY KEY,
    endpoint TEXT UNIQUE NOT NULL,
    keys_json TEXT NOT NULL,           -- JSON: {"p256dh": "...", "auth": "..."}
    created_at TEXT NOT NULL
);
```

**`parse_status` field:** Enables "save immediately, parse later." Meal entries are saved with `parse_status='pending'`, Claude parses in the background, updates to `'parsed'` or `'failed'`. Failed entries are retried with exponential backoff (1min, 5min, 30min, then stop).

**`parsed_ingredients` structure:**
```json
{
  "confirmed": ["rice", "chicken", "bok choy"],
  "likely": ["soy sauce", "garlic"],
  "source": "text"
}
```

**`ingredient_aliases` table:** Maps variant strings to canonical terms (e.g., `tamari → soy sauce`). Starts empty, populated manually as inconsistencies are noticed. The correlation algorithm resolves through this table before computing frequencies — corrections apply retroactively.

## API Endpoints

| Method | Path | Behaviour |
|---|---|---|
| `POST` | `/api/log` | Save entry immediately. If meal, queue Claude parsing as background task. Returns entry ID. |
| `GET` | `/api/logs` | List entries. Filters: `?type=`, `?from=`, `?to=` |
| `POST` | `/api/analyse` | Kick off async analysis. Returns job ID. |
| `GET` | `/api/analyse/{job_id}` | Poll for analysis result (stats table + Claude summary). |
| `POST` | `/api/barcode/{upc}` | Open Food Facts lookup → ingredient list |
| `POST` | `/api/push/subscribe` | Register Web Push subscription |
| `GET` | `/api/admin/ingredients` | All unique ingredients sorted by frequency |
| `POST` | `/api/admin/aliases` | Add/update alias mapping |
| `PUT` | `/api/log/{id}` | Update an existing log entry |
| `DELETE` | `/api/log/{id}` | Delete a log entry |

**Error format:** All errors return `{"error": "message", "detail": "..."}` with appropriate HTTP status codes.

**Analysis job storage:** Job state and results stored in an in-memory dict keyed by job ID. Jobs are short-lived (analysis completes in seconds) so persistence isn't needed — if the server restarts, the user just re-runs the analysis.

**Security note:** The admin endpoints have no auth. This is acceptable while the app is single-user on Tailscale. **TODO: If this app is ever exposed beyond Tailscale, all endpoints — especially `/api/admin/*` and `/api/log` — need authentication. Without it, anyone with network access can read health data and write to the database.**

Brief success toast (~2 seconds), fields clear, stay on same tab. No page navigation.

## Offline Strategy

- Service worker caches app shell so the app loads offline
- When offline, `POST /api/log` calls intercepted and stored in IndexedDB (including photo blobs)
- Non-blocking banner: "You're offline — entry queued and will sync when you're back online"
- When connectivity returns, service worker replays queued entries in order
- Synced entries get `synced` flag set to `1` on server

## Web Push Notifications

- On first app load, prompt to allow notifications
- Backend uses `pywebpush` — no Firebase dependency
- Scheduled via APScheduler (configurable in `config.py`):
  - Breakfast: 7:30am AEST
  - Lunch: 12:00pm AEST
  - Dinner: 6:30pm AEST
- Skips notification if a meal was logged in the preceding 2 hours
- Notification text only ("Time to log lunch") — no health data leaves the server

## Correlation Algorithm

Deterministic code (pandas/scipy). Claude's role is summarising the output only.

**Steps:**
1. Resolve all ingredient names through `ingredient_aliases` table
2. For each flare event, define pre-flare window: meals in the 6–48h before the flare timestamp
3. Collect ingredients from meals in flare windows → flare ingredient set (with counts)
4. Collect ingredients from meals outside any flare window → baseline ingredient set (with counts)
5. Per ingredient, compute:
   - `flare_freq` = count in flare windows / total ingredients in flare windows
   - `baseline_freq` = count in baseline / total ingredients in baseline
   - `lift` = `flare_freq / baseline_freq` (infinity if baseline_freq == 0)
   - `flare_appearances` = number of distinct flare events this ingredient appeared in
6. Flag confounded flare events: medication within ±12h of flare
7. Filter to ingredients with ≥2 flare appearances
8. Sort by lift descending
9. Pass structured results to Claude for plain-English summary

**`use_likely` toggle:** Default off. When on, includes `likely` ingredients alongside `confirmed`.

**Warning threshold:** <10 flare events → prominent warning.

## Claude API Integration

Three roles, all using prompts stored as constants in `prompts.py`:

| Role | Trigger | Model | Failure handling |
|---|---|---|---|
| Text parsing | Meal logged | Configurable (start Haiku) | Entry saved immediately, `parse_status='failed'`, retry with backoff |
| Image parsing | Future (A/B test) | Configurable | Same as text parsing |
| Analysis summary | After correlation | Haiku | Analysis stats still shown, summary shows error message |

**Retry strategy for parsing failures:** Exponential backoff — 1min, 5min, 30min, then stop. Admin ingredients page shows count of unparsed entries.

**Prompts:** Three prompts from the handover document (text parsing, image parsing, analysis summary), stored verbatim in `prompts.py`.

## Image Pipeline — Future A/B Testing

Meal photos stored from day 1 alongside text-parsed entries. Future workflow:
1. Take entries with both photo and text-parsed ingredients
2. Run Claude vision API on each photo
3. Compare ingredient lists side by side
4. Test Haiku vs Sonnet for accuracy/cost tradeoff
5. Once satisfied, add image as alternative input path in PWA

No changes to correlation algorithm required — same schema regardless of source.

## Backup Strategy

Two-tier:

**Tier 1 — Local (Raspberry Pi + external SSD):**
Laptop rsyncs SQLite + images to Pi on cron schedule. Two physical copies on home network.

**Tier 2 — Offsite encrypted (Backblaze B2, optional):**
`rclone crypt` encrypts locally before upload. Backblaze never sees plaintext.

Backup script runs via cron (`0 3 * * *`), uses `sqlite3 .backup` for consistent snapshots, keeps 30 days of local DB snapshots.
